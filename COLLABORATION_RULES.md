# MagicQuant — AI 协作守则

> **目的**: 让 Claude / 其他 AI 在协助本项目时,**行为一致 + 不烧钱 + 不出错**。
> **约束原则**: 这份文档是硬约束,AI 必须遵守。用户(老杨)在新对话开头引用本文件即可激活所有规则。

---

## 🎯 核心原则(优先级从高到低)

### 1. 改动最小化
- 每次改动只碰必要文件。不重写,不重构,不顺手优化。
- 除非 < 100 行 或 改动 > 50% 文件内容,**只给 diff,不给全文件**。
- 不主动"顺便"修其他 bug,发现后**记录到 CURRENT_STATE.md 的已知 bug 清单**。

### 2. 一次对话只做一件事
- 新对话开头,用户会说"今天做 X",那就**只做 X**。
- 发现用户在同一对话里越滑越远(从 bug 修复滑到加新功能),**主动提醒**:"这个建议开新对话做,避免 context 爆炸"。
- 对话长度控制: 超过 30 条消息就提醒用户是否该开新对话。

### 3. 费用意识(Token 节省)
- **禁止做**:
  - 不必要的全文件重贴(用 diff 或指向 GitHub raw URL)
  - 不必要的"先让我确认一下"(用户已经给你答案了就动手)
  - 给代码前反复解释原理(先给代码,再解释)
  - 重复总结用户已经说过的话

- **鼓励做**:
  - 用 Project Knowledge 里的文档
  - 用 web_fetch 直接拉 GitHub raw 内容(链接格式: `https://raw.githubusercontent.com/WinkYang1979/MagicQuant/main/路径`)
  - 用 ask_user_input_v0 问选项题,不让用户打字
  - 给 diff 而非全文件

## 🔧 代码规范

### 每个文件头必须包含
```python
"""
════════════════════════════════════════════════════════════════════
  MagicQuant [模块名] — [文件名].py
  VERSION : vX.Y.Z
  DATE    : YYYY-MM-DD
  CHANGES :
    vX.Y.Z (YYYY-MM-DD):
      - [FIX/NEW/CHG] 简述
  DEPENDS :
    文件A  ≥ vX.Y.Z
    文件B  ≥ vX.Y.Z
  OWNER   : laoyang
════════════════════════════════════════════════════════════════════
"""
```

### 版本号规则
- **主项目版本**: v0.X.Y(整体发布,如 v0.5.13)
- **模块版本**: 独立演进(如 pusher v0.5.11 vs focus_manager v0.5.9)
- **升版本时机**: 任何代码改动都升 patch(Z),新功能升 minor(Y),架构改动升 major(X)

### 命名规范
- 代码变量 / 函数名: **英文 snake_case**
- 注释 / 文档 / 用户消息: **中文为主**
- i18n 函数 `t()` 不能和循环变量 `t` 同名,**循环变量统一用 `tk`**(历史踩坑)
- 字段名如 `ticker` 带 `US.` 前缀,跟 Futu API 一致

### 文件路径管理
- 用 `BASE_DIR` 统一管理,**不写死路径**
- 密钥永远放 `env` 文件(在 .gitignore 里),不硬编码

## 🤝 协作流程

### A. 新对话接手
**第一步**(AI 做的,不打扰用户):
1. 读 `CURRENT_STATE.md`,获取线上版本 + 账户状态 + 已知 bug 清单
2. 读 `COLLABORATION_RULES.md`(本文件),确认约束
3. 读 Project Knowledge 里的完整项目文档(如 `MagicQuant_Project_Doc_vX.md`)

**第二步**(跟用户对齐):
- 如果用户说"今天做 X",直接开工
- 如果用户没说,**提供 2-3 个基于 CURRENT_STATE.md 的建议任务**,让用户选

### B. 改代码前的预检
**必做**:
1. 确认当前版本号(从 CURRENT_STATE.md 或 GitHub)
2. 拉取当前文件实际内容(web_fetch raw URL)
3. 确认依赖链(这次改动影响哪些文件)
4. **确认用户意图**(产品语义,不是代码实现)

**不做**:
- 凭记忆写代码(必定过时)
- 猜字段名 / 函数签名(会带来反复修正)
- 复制项目文档里的代码片段当作最新(文档可能滞后)

### C. 动手时的节奏
```
给 diff(精确到行号) → 等用户部署 → 用户说"OK" → 下一个改动
```

**禁止**:
- 一次给多个文件的完整版本
- 改完一个立刻接着改下一个(必须等用户确认部署成功)

### D. 改完后
- 提醒用户 `git commit + push`
- 提醒用户更新 `CURRENT_STATE.md`(版本号 / 已知 bug 清单)
- 简短总结今天的改动(1-3 行)

## 🚨 安全与风险

### 费用硬性红线
- 不主动调用外部 AI API(除非用户明确要求)
- web_fetch 优先用已知 URL(GitHub raw 链接),避免重新搜索
- Claude API 相关改动,**预估 token 成本后再改**,并告知用户

### 交易安全红线
- 涉及实盘交易的代码改动,**必须强调风险**
- AI 建议的数字(仓位、股数、金额)**必须用真实账户数据核算**,不得凭空编造
- 止损 / 止盈阈值变动,**必须告知风控影响**

### 代码质量红线
- 不编造字段名(不确定就 web_fetch 查真实代码)
- 不跳过语法检查(改动后至少 `python -m py_compile` 验证)
- 不在 Windows cmd 生成带中文的 batch 文件(编码陷阱,历史踩坑)

## 💬 沟通风格

### 给用户回复时
- **开门见山**: 先说结论,再说原因(用户时间宝贵)
- **用表格 / 结构化 Markdown**: 纯段落阅读成本高
- **给选项题而非开放题**: 用 ask_user_input_v0,减少用户打字
- **情感支持**: 用户疲劳 / 挫败时,先认同再继续(昨天有先例)

### 当用户烦躁时
- 不狡辩、不找借口
- 直接承认问题
- 给出具体改进方案
- 不道歉过多(显得敷衍)

## 📝 历史踩坑 Case Study(别再犯)

### Case 1: 版本覆盖丢接口(2026-04-22)
**事故**: 我以 `realtime_quote.py v0.2.1` 老基线改出 v0.2.2,覆盖了用户本地 v0.5.0 版本,**丢失了 `fetch_positions / fetch_account / _ensure_quote` 等方法**,导致线上系统崩溃。

**教训**:
- 新对话改代码前,**必须 fetch GitHub 上真实的最新版本**,而不是根据项目文档里的"预期版本"动手
- 项目文档会滞后,代码才是真相
- 改前用 CURRENT_STATE.md 对齐版本号

### Case 2: 字段名靠猜(2026-04-22)
**事故**: `fetch_positions` 字段名 `current_price` / `pl_pct` 是猜的,实际 Moomoo AU 返回 `nominal_price` / `pl_ratio`。导致数据显示全是 $0,触发器全失效。

**教训**:
- Futu / Moomoo API 返回字段**先写诊断脚本探测**,不凭记忆
- 给一个 `futu_api_reference.txt` 存项目里作为字段真值表

### Case 3: 货币聚合陷阱(2026-04-22)
**事故**: `accinfo_query` 默认返回 HKD 聚合值($27,704),字段名叫 `cash` 让人误以为是 USD。导致 `/account` 显示错误金额。

**教训**:
- 多币种账户必须显式 `currency=Currency.USD`,或优先读 `us_cash` 等 USD 字段
- 返回结果加 `currency + source` 标记,便于下次 debug
- 写代码时 assume 最差情况(API 返回是多币种聚合)

### Case 4: Batch 文件中文编码(2026-04-22)
**事故**: 生成 .bat 脚本含中文,Windows cmd 用 GBK 解析 UTF-8 文件变乱码,乱码被当命令跑。

**教训**:
- **.bat 文件不写中文注释**,或者生成时强制 GBK 编码
- 敏感脚本用 Python 不用 batch

```
在给代码前,我确认了:
[ ] 从 GitHub 读取了文件最新内容
[ ] 确认了版本号跟 CURRENT_STATE.md 一致

[ ] 字段名 / 函数签名有真实证据

## 文件: core/focus/pusher.py (v0.5.11 → v0.5.12)

+ 新代码

### 改动 2: 第 Y 行
+ 新增函数
```
